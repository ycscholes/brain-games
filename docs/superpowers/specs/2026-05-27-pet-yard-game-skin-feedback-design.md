# 宠物小院绘本游戏风换肤与反馈设计

## 背景

当前宠物小院页面已经具备领养、切换宠物、喂食、饱食度、积分余额和死亡/纪念等核心功能，但视觉表达偏卡片式管理面板。用户希望页面更有游戏感，同时本轮暂不做首屏庭院式大布局重排。

本设计将本轮目标收敛为：在保留现有信息架构和底层养成规则的前提下，统一绘本游戏风资产，并强化喂食、抚摸、切换宠物的即时反馈。

## 决策记录

- 风格方向：养成小院，而不是像素街机或轻 RPG 经营。
- 实施路线：保留当前结构，做游戏化换肤和反馈增强。
- 反馈范围：情绪气泡、状态台词、喂食奖励反馈。
- 资产范围：完整 UI 资产包，包括宠物、食物、背景、按钮、状态牌、商店图标等。
- 暂不做：首屏一眼像游戏的庭院主场景布局重排，记录为后续 todo。

## 目标

1. 操作像游戏：喂食、抚摸、切换宠物都有明确、短暂、可感知的视觉反馈。
2. 资产统一：宠物、食物、按钮、状态牌、商店视觉属于同一套绘本游戏风。
3. 规则稳定：不改变积分经济、饱食衰减、领养成本、死亡状态、云同步和本地存储规则。
4. 范围可控：主要改展示层和资产配置，不引入任务、装饰、升级奖励等新系统。

## 非目标

- 不重构首屏为完整庭院主场景。
- 不新增宠物养成数据字段。
- 不新增装饰系统、任务系统、成就系统或背包系统。
- 不调整 `src/utils/petStorage.ts` 中的经济和生命周期规则。
- 不重做首页、训练页或其他游戏页面。

## 用户体验设计

### 视觉语言

页面从玻璃拟态卡片转为绘本游戏 UI：

- 大容器使用纸张、木牌、布告栏、食物篮等拟物化表达。
- 操作按钮使用饱满、可点击的游戏按钮样式，具备按压反馈。
- 状态牌使用木牌或小标签表达，例如“状态稳定”“需要喂食”“已离开”。
- 商店列表从普通列表转为食物货架或食物篮卡片。
- 颜色以自然绿、暖黄、木色、天空蓝为主，避免单一紫蓝管理后台感。

### 资产包

完整绘本游戏风资产包应覆盖：

- 宠物：5 种皮肤 x 3 种状态，共 15 张 PNG。
- 食物：苹果、鲜鱼、大牛排，共 3 张图标。
- 场景：小院背景或局部背景元素，可用于当前宠物展示区和空状态。
- UI：按钮底板、状态牌、商店/食物区域图标或纹理。

资产规格：

- 宠物图保持透明背景 PNG，文件命名兼容现有 `resolvePetSpriteUrl` 规则。
- 食物和 UI 图标优先使用透明 PNG，便于在 Taro 小程序中组合。
- 小院背景可使用 CSS 渐变加局部图片，也可使用单张背景图；实现时以包体和渲染稳定性为准。
- 资产需要在小尺寸下可辨认，尤其是宠物切换 chip 和食物商店行。

资产生成规则：

- 图片生成任务必须新建 subagent 执行，避免主实现流程被长时间资产迭代阻塞。
- 图片生成模型必须使用 `gpt-image-2`。
- subagent 交付物需包含生成提示词、产物路径、命名映射和人工筛选说明。
- 主实现流程只接入已确认的资产文件；不在页面实现步骤中临时生成或重命名图片。

### 情绪气泡与状态台词

在当前宠物展示区域增加本地展示状态：

- `idle`：展示稳定陪伴类短句。
- `hungry`：展示提醒喂食类短句。
- `feed`：喂食成功后展示吃饱或感谢类短句。
- `cuddle`：点击宠物或切换宠物后展示亲密回应类短句。
- `dead`：展示纪念/告别类短句，避免喂食和抚摸反馈。

台词只用于展示，不落库，不影响同步。

### 操作反馈

喂食成功：

- 宠物进入 `feed` 动作。
- 展示短暂食物图标或喂食奖励浮层。
- 展示 `+饱食度` 和 `-积分` 反馈。
- 气泡台词切换为喂食成功短句。

抚摸宠物：

- 点击宠物时进入 `cuddle` 动作。
- 展示亲密台词。
- 不改变任何数据。

切换宠物：

- 当前宠物切换后播放欢迎或状态台词。
- 选中态更像游戏中的角色选择，而不是普通列表 chip。

失败反馈：

- 积分不足、已离开宠物不可喂食等失败情况继续使用 toast。
- 可在当前页面同步显示轻量错误台词，但不得替代系统 toast。

## 技术设计

### 主要改动区域

- `src/pages/pet/index.tsx`：增加本地反馈状态、台词选择、喂食奖励展示、交互触发。
- `src/pages/pet/index.scss`：绘本游戏风换肤、按钮/状态牌/商店/宠物区域样式。
- `src/pages/pet/components/PetSprite/index.tsx`：保持现有接口，必要时增强 className 或加载状态展示。
- `src/pages/pet/components/PetSprite/index.scss`：匹配新资产的阴影、光效、动作幅度。
- `src/config/remoteAssets.ts` 或相关资产配置：接入新宠物和 UI 资产，保持旧路径兼容。
- `src/pages/pet/types.ts`：仅在需要补充展示常量时调整，不新增持久化数据字段。

### 本地反馈状态

页面组件维护短暂 UI 状态，例如：

- 当前宠物动作：沿用 `petMotion`。
- 当前气泡台词：由当前宠物状态和最近操作推导。
- 喂食反馈：记录食物、恢复值、消耗积分和短暂可见状态。

所有 timer 在组件卸载时清理，避免切页后继续 setState。

### 数据流

1. 页面读取现有 `PetStorageData`。
2. 用户执行喂食、领养、切换宠物。
3. 成功的数据操作仍调用现有 `feedPet`、`adoptPet` 或本地 activePetId 更新。
4. 展示层根据操作结果触发本地反馈。
5. 定时器结束后反馈回到当前宠物的基础状态文案。

### 错误处理

- 图片加载失败继续使用 `PET_SKIN_EMOJI` 兜底。
- 远程资产不可用时页面结构和操作必须可用。
- 反馈状态必须能被新操作覆盖，避免旧 timer 把新反馈提前清掉。
- 已离开宠物不触发喂食和抚摸的正向反馈。

## 验收标准

1. 喂食成功后能看到宠物动作、食物/数值反馈、状态台词变化。
2. 点击宠物能看到抚摸反馈和亲密台词。
3. 切换宠物时选中态明确，当前宠物有短暂回应。
4. 宠物、食物、按钮、状态牌、商店区域属于统一绘本游戏风。
5. 积分余额、饱食度、领养成本、死亡状态与现有逻辑一致。
6. 图片加载失败时仍有 emoji 兜底，不影响领养、喂食和切换。
7. 现有宠物相关单元测试通过。

## 测试计划

- 运行宠物存储相关测试：`npm test -- tests/unit/petStorage.test.ts`。
- 如改动共享数据同步触发，补充运行：`npm test -- tests/unit/cloudSyncMeta.test.ts`。
- 手动验证微信小程序页面：
  - 无宠物时显示领养入口和统一风格空状态。
  - 有存活宠物时可抚摸、喂食、切换。
  - 饥饿宠物显示饥饿台词和状态牌。
  - 已离开宠物进入纪念态，喂食不可用。
  - 资产加载失败时显示兜底。
